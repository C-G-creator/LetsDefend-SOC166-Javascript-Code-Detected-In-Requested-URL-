# SOC166 — JavaScript Code Detected in Requested URL

In this post, I’ll walk through my investigation of SOC166 — JavaScript Code Detected in Requested URL, an alert generated by LetsDefend. The goal is not only to explain what happened, but to show the investigative process behind the conclusion, from initial triage to final assessment.

## Alert Overview

On February 26, 2022, at 6:56 PM, LetsDefend generated an alert for JavaScript code detected in a requested URL. The targeted asset was WebServer1002 with the internal IP address 172.16.17.17.

The key alert details were:

- Requested URL: `hxxps://172.16.17.17/search/?q=<$script>javascript:$alert(1)<$/script>`
- HTTP method: `GET`
- Source IP: `112.85.42[.]13`

The presence of `script` and `alert` in the URL immediately suggested a possible cross-site scripting (XSS) attempt. While those keywords alone do not prove exploitation, they are common indicators in reflected XSS payloads and are worth investigating closely.

## Initial Triage

My first step was to confirm that `172.16.17.17` belonged to the target web server and that the activity was not expected application behavior. Once that was verified, I reviewed the source IP address using VirusTotal and AbuseIPDB to determine whether it had any prior malicious reputation.

The source IP, `112.85.42[.]13`, had multiple AbuseIPDB reports associated with brute force activity. VirusTotal showed a community score of `-15`, which reinforced suspicion, even though the public score itself was listed as `0`. That combination suggested the IP had a poor reputation and should be treated as untrusted.

## Log Analysis

Next, I moved into Log Management and reviewed events for the targeted host on February 26, 2022. I found eight events tied to the alert, including the event that triggered the detection rule.

The sequence told a clear story:

- 6:34 PM: A permitted GET request was made to a URL containing the target host.
- 6:35 PM: Another request targeted `hxxps://172.16.17.17/about-us/`, suggesting reconnaissance.
- 6:45 PM: A request containing a test search term appeared, which looked like probing behavior.
- Later events: The attacker gradually shifted toward script injection attempts.
- Final event: The rule triggered on the eighth request after multiple suspicious URL patterns were observed.

A few details stood out during review. Events 4 through 8 all returned HTTP status code `302`, and the response size was `0`. That matters because it indicates the requests were redirected and did not receive a meaningful response body. In this case, that strongly suggested the attempted payloads were not successfully executed.

## Threat Assessment

At this point, the evidence pointed to a likely reflected XSS attempt. The source IP had a questionable reputation, the URL patterns matched common XSS indicators, and the request sequence showed escalating probing behavior before the final payload attempt.

It is important to separate suspicion from successful infiltration. The traffic was clearly malicious in intent, but the available evidence did not show that the attack achieved execution on the server. The repeated `302` responses and empty response sizes supported that conclusion.

Before applying the playbook, I re-checked the timeline and supporting evidence to make sure the conclusion was based on observable data rather than assumption. That extra validation is important in SOC work because it can prevent overcalling an incident or missing a real escalation path. ### Step 1: Determine Whether the Traffic Is Malicious

Yes. The source IP had prior malicious reporting, and the request pattern aligned with XSS behavior. The activity was not normal user traffic.

### Step 2: Identify the Attack Type

This was assessed as a cross-site scripting attempt, likely reflected XSS. The payload structure and keyword usage in the URL were consistent with that classification.

### Step 3: Determine Whether This Was a Planned Test

There was no evidence of a sanctioned security test. No internal notice, change request, or testing coordination was identified in the logs or surrounding context.

### Step 4: Identify Traffic Direction

The traffic flowed from external to internal, with the source attempting to interact with the web server through crafted requests.

### Step 5: Determine Whether the Attack Succeeded

Based on the logs, the attack did not appear to succeed. The final requests returned `302` redirects and `0-byte` responses, which suggested the payload did not execute as intended.

### Step 7: Escalation Decision

Based on the evidence available, this incident did not require Tier 2 escalation. The attack was suspicious and malicious in nature, but it appeared to be unsuccessful and contained at the web request layer.

## Conclusion

On February 26, 2022, LetsDefend generated an alert for JavaScript code detected in a requested URL against `172.16.17.17`. After reviewing the request pattern, source reputation, and log behavior, I determined that the source IP `112.85.42[.]13` was attempting a cross-site scripting attack.

The attack showed a clear progression from reconnaissance to payload delivery, but the repeated `302` redirects and `0-byte` responses indicated that the attempt was unsuccessful. From a SOC perspective, this was a good example of how careful log review and traffic analysis can distinguish between malicious intent and confirmed compromise.
